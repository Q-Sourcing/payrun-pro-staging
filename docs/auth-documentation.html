1. System Overview

Q-Payroll uses Supabase Auth as its identity provider combined with a custom Role-Based Access Control (RBAC) system built on PostgreSQL. Authentication is stateless JWT-based; authorization is enforced at three layers:

  • PostgreSQL Row-Level Security (RLS)
  • React frontend permission gates
  • Supabase Edge Function guards

The system supports multiple tenants (organizations), each with isolated roles and permissions. A single Supabase project serves all tenants; tenant isolation is enforced via org_id columns on RBAC tables and RLS policies.

2. Authentication Flow

2.1 Invitation Flow (New User)

  1. Admin sends invite — Admin calls the invite-user Edge Function (POST /functions/v1/invite-user?action=invite) with { email, full_name, role, role_data: { module_access } }.
  2. Supabase creates auth user — supabase.auth.admin.inviteUserByEmail() creates an unconfirmed auth user and sends a magic-link email.
  3. Invitation record created — A row is inserted into user_management_invitations with: email, role, role_data, token (UUID), status: 'pending', expires_at (48 hours).
  4. User profile pre-created — A row is upserted into user_profiles with status: 'pending'.
  5. Email delivered — The email link redirects to /set-password?token=<custom-token>. The Supabase magic link appends #access_token=...&refresh_token=...&type=invite as a URL hash.

2.2 Password Creation Flow (/set-password)

The SetPassword.tsx page handles three URL patterns:

  • Path A — Supabase hash tokens: URL contains #access_token=...&type=invite. Page calls supabase.auth.setSession() to establish a session, then supabase.auth.updateUser({ password }) to set the password. After success, calls the accept action on invite-user with { token, user_id }, then supabase.auth.signOut() and redirects to /login.
  • Path B — PKCE code: URL contains ?code=.... Page calls supabase.auth.exchangeCodeForSession(code).
  • Path C — Custom token only: URL contains ?token=... but no hash. Page calls invite-user?action=accept with { token, password } (server-side password set).

2.3 Invitation Accept Action (Edge Function)

When invite-user?action=accept is called with { token, user_id }:

  1. Fetches invitation record, validates token and expiry.
  2. Marks invitation status: 'accepted', accepted_at: now().
  3. Determines org_id: Looks up user_profiles.organization_id; defaults to '00000000-0000-0000-0000-000000000001' (the primary org).
  4. Upserts user_profiles with { status: 'active', organization_id: orgId, ... }.
  5. Deletes existing org-level assignments (except PLATFORM_SUPER_ADMIN).
  6. Inserts rbac_assignment with { user_id, role_code: inv.role, scope_type: 'GLOBAL', org_id }. This fires the sync_rbac_to_jwt trigger.
  7. Inserts rbac_grants for each module in role_data.module_access.
Critical: organization_id must be set in user_profiles before the rbac_assignment INSERT. The validate_rbac_assignment trigger checks user_profiles.organization_id = assignment.org_id and will silently block the insert if this condition is not met.

2.4 Login Flow

  1. User submits email/password to ModernLoginForm.tsx → supabase.auth.signInWithPassword().
  2. Supabase returns a signed JWT containing app_metadata with RBAC claims.
  3. AuthProvider.tsx hydrates session: calls hydrateFromSession(s) first (synchronous, reads JWT), sets isLoading: false, then runs checkAccountActive(s) as a non-blocking background check.
  4. UserContext is populated from JWT claims. No additional DB queries needed for permission checks at render time.
Note on hydration order: hydrateFromSession and setIsLoading(false) must run before checkAccountActive. The active-check is a slow RLS query that used to block hydration and cause 10-second timeouts redirecting users back to login.

2.5 Post-Login Session Handling

  • JWT is stored in localStorage as sb-<project_ref>-auth-token.
  • onAuthStateChange listener in AuthProvider re-hydrates session on token refresh.
  • Token auto-refresh is handled by the Supabase client.
  • Permission changes take effect immediately for new logins (via JWT sync triggers). Existing sessions see updates on next token refresh or re-login.

3. JWT Structure

The JWT app_metadata object contains:

{
  "provider": "email",
  "providers": ["email"],
  "rbac_roles": [
    {
      "role": "ADMIN",
      "scope_type": "GLOBAL",
      "scope_id": null,
      "org_id": "00000000-0000-0000-0000-000000000001"
    }
  ],
  "rbac_permissions": [
    "people.view",
    "people.create",
    "payroll.view",
    "..."
  ],
  "is_platform_admin": false,
  "organization_id": "00000000-0000-0000-0000-000000000001"
}

Platform super admin JWT (different structure):

{
  "rbac_roles": [
    { "role": "PLATFORM_SUPER_ADMIN", "scope_type": "GLOBAL", "org_id": "00000000-0000-0000-0000-000000000000" }
  ],
  "rbac_permissions": [],
  "is_platform_admin": true,
  "organization_id": "00000000-0000-0000-0000-000000000000"
}

Note: PLATFORM_SUPER_ADMIN has no entries in rbac_role_permissions — it relies solely on is_platform_admin: true. All permission checks in the frontend short-circuit when this flag is true.

4. RBAC Database Schema

4.1 Tables

rbac_roles — Tenant-scoped role definitions

org_id      UUID NOT NULL FK→organizations(id) ON DELETE CASCADE
PRIMARY KEY (code, org_id)

role_code      TEXT NOT NULL
permission_key TEXT NOT NULL FK→rbac_permissions(key)
org_id         UUID NOT NULL FK→organizations(id)
PRIMARY KEY (role_code, permission_key, org_id)
FK: (role_code, org_id) → rbac_roles(code, org_id)

rbac_assignments — Maps users to roles

id          UUID PRIMARY KEY
user_id     UUID NOT NULL FK→auth.users(id) ON DELETE CASCADE
role_code   TEXT NOT NULL
scope_type  TEXT  -- 'GLOBAL' | 'ORGANIZATION' | 'COMPANY' | 'PROJECT' | 'SELF'
scope_id    UUID NULLABLE
org_id      UUID NULLABLE FK→organizations(id)
assigned_by UUID FK→auth.users(id)
UNIQUE INDEX: (user_id, role_code, scope_type, COALESCE(scope_id, '00000000-0000-0000-0000-000000000000'))
FK: (role_code, org_id) → rbac_roles(code, org_id)

rbac_grants — User-level permission overrides

id             UUID PRIMARY KEY
user_id        UUID NULLABLE FK→auth.users(id)
role_code      TEXT NULLABLE FK→rbac_roles(code)
permission_key TEXT NOT NULL FK→rbac_permissions(key)
scope_type     TEXT  -- 'ORGANIZATION' | 'COMPANY' | 'PROJECT'
scope_id       UUID NOT NULL
effect         TEXT  -- 'ALLOW' | 'DENY'
reason         TEXT
granted_by     TEXT
valid_until    TIMESTAMPTZ NULLABLE
CONSTRAINT: either user_id OR role_code must be set (not both null)
UNIQUE INDEX (user_id, permission_key, scope_type, scope_id) WHERE user_id IS NOT NULL
UNIQUE INDEX (role_code, permission_key, scope_type, scope_id) WHERE role_code IS NOT NULL

5. Permission Sync System (Triggers)

PostgreSQL triggers keep JWT claims in sync with DB state so permission changes take effect without requiring re-login.

5.1 sync_rbac_to_jwt (on rbac_assignments)

Fires AFTER INSERT/UPDATE/DELETE. For the affected user_id:

  • Sets raw_app_meta_data.rbac_roles — array of all current assignments
  • Sets raw_app_meta_data.rbac_permissions — UNION of (all role-level permissions) + (user-level ALLOW grants)
  • Sets raw_app_meta_data.is_platform_admin — true if any assignment is PLATFORM tier + GLOBAL scope
  • Sets raw_app_meta_data.organization_id — from the user's first non-PLATFORM assignment

5.2 sync_grants_to_jwt (on rbac_grants)

Fires AFTER INSERT/UPDATE/DELETE. For the affected user_id:

  • Re-syncs raw_app_meta_data.rbac_permissions (role permissions UNION user grants)

5.3 trg_sync_role_perms_to_jwt (on rbac_role_permissions)

Fires AFTER INSERT/UPDATE/DELETE. For every user currently assigned to the affected role:

  • Re-syncs raw_app_meta_data.rbac_permissions for each affected user
Why this matters: Without this trigger, changes made in the Roles & Permissions UI (toggling permissions on/off) only took effect when affected users logged out and back in. This trigger was the missing piece that enables real-time permission propagation.

5.4 validate_rbac_assignment (on rbac_assignments, BEFORE INSERT/UPDATE)

Guards against invalid scoping. Validates:

  • User's user_profiles.organization_id matches NEW.org_id
  • COMPANY scope: scope_id must belong to the org
  • PROJECT scope: scope_id must belong to the org
  • ORGANIZATION scope: scope_id must equal org_id

5.5 trg_provision_default_roles (on organizations)

Fires AFTER INSERT on organizations. Calls provision_default_roles(NEW.id). Skips the sentinel org (00000000-...0000). Provisions all 7 default roles (ADMIN, HR, GM, FINANCE, STAFF, SELF_USER, SELF_CONTRACTOR) with sensible default permission sets for any newly created tenant.

7. RLS Policies

7.1 check_is_org_super_admin(user_id uuid)

The core authorization function used as USING/WITH CHECK expression for all write operations on RBAC tables:

CREATE OR REPLACE FUNCTION public.check_is_org_super_admin(user_id uuid)
RETURNS boolean LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
BEGIN
  IF EXISTS (
    SELECT 1 FROM public.rbac_assignments
    WHERE rbac_assignments.user_id = user_id
      AND role_code IN ('PLATFORM_SUPER_ADMIN', 'ORG_ADMIN', 'ADMIN', 'GM')
  ) THEN RETURN TRUE; END IF;
  RETURN FALSE;
END; $$;
Important: This function previously referenced public.users (a dropped table), which caused PostgREST to return HTTP 404 (not 403) for all write operations — a misleading error. If this function ever throws a SQL exception, PostgREST will return 404 for the entire request, not just a permission error.

8. Frontend Permission Model

8.1 AuthProvider Hydration Order

// CORRECT order (prevents 10-second timeout):
hydrateFromSession(session)  // synchronous, reads JWT claims immediately
setIsLoading(false)           // unblocks UI immediately — does NOT wait for DB
checkAccountActive(session)  // async RLS query, runs in background

8.2 UserContext Structure

interface UserContext {
  id: string;
  email: string;
  roles: Array<{
    role: string;
    scope_type: 'GLOBAL' | 'ORGANIZATION' | 'COMPANY' | 'PROJECT' | 'SELF';
    scope_id: string | null;
    org_id: string;
  }>;
  permissions: string[];         // from app_metadata.rbac_permissions
  isPlatformAdmin: boolean;      // from app_metadata.is_platform_admin
  organizationId: string | null; // from app_metadata.organization_id
  isImpersonated: boolean;
}

8.3 isOrgAdmin Function

Used by the Sidebar and page guards to determine whether to show admin-only sections (Settings, EHS, Assets):

export function isOrgAdmin(ctx: UserContext | null): boolean {
  if (!ctx) return false;
  return ctx.isPlatformAdmin || ctx.roles.some((r) =>
    r.role === 'ORG_ADMIN' || r.role === 'ADMIN' || r.role === 'GM'
  );
}
Must include both the legacy code ORG_ADMIN and actual DB codes ADMIN and GM. Missing either causes Settings/EHS/Assets to be invisible for those users.

9. Permissions Registry

9.1 Single Source of Truth

src/lib/constants/permissions-registry.ts defines all modules and their permission keys. The Roles & Permissions UI auto-discovers modules from this registry — no UI changes are needed when adding new modules.

9.2 Adding a New Module — Complete Procedure

  1. Add module definition to SYSTEM_MODULES_REGISTRY in permissions-registry.ts (UI auto-discovers it)
  2. Write a migration to register the keys in the DB: 
    INSERT INTO public.rbac_permissions (key, category, description)
VALUES ('mymodule.view', 'My Module', 'View my module'), ...
ON CONFLICT (key) DO NOTHING;
  3. Add the module's keys to MODULE_PERMISSION_MAP in supabase/functions/invite-user/index.ts
  4. Optionally add keys to provision_default_roles() for auto-assignment to new tenants
  5. Add a sidebar visibility check in MainLayout.tsx

10. Edge Functions

10.1 invite-user

The primary user lifecycle function. Auth check: caller must have ADMIN, HR, or PLATFORM_SUPER_ADMIN role.

ActionDescription
inviteCreates auth user, invitation record, pre-creates profile
acceptActivates user: sets profile org_id, assigns RBAC role, writes grants

Origin detection: Reads the origin header to build redirect URLs, supporting localhost development, *.flipafrica.app, and *.lovable.app without hardcoded URLs.

10.2 manage-users

Administrative user management (CRUD on assignments). Auth check: same as invite-user.

  • GET — lists users with their current role assignments
  • POST — creates a new rbac_assignment (role change)
  • PATCH — updates role and/or module grants for a user
  • DELETE — removes user from the system

10.3 invite-org-user

Organization-scoped variant of user invitation for project/department-level onboarding. Fixed a temporal dead zone bug where inviteToken was used before declaration.

10.4 admin-cleanup-invite / cleanup-invited-users

Administrative cleanup functions for stale invitations. Migrated off the legacy user_management_profiles and user_invites tables.

11. Known Issues, Root Causes, and Fixes Applied

Issue 1 — check_is_org_super_admin referenced dropped table

Root cause: The function body contained SELECT 1 FROM public.users after the public.users table was dropped.

Symptom: PostgREST returned HTTP 404 (not 403) for ALL write operations on RBAC tables. This is the behavior PostgREST exhibits when an RLS function raises a SQL exception — misleading because it looks like a "not found" rather than a permissions error.

Fix: Removed the legacy users table reference; function now only queries rbac_assignments.

Issue 2 — Invited ADMIN users saw only "My Dashboard"

Root cause: Two-part failure: (a) The invite-user accept action upserted user_profiles without organization_id. (b) The validate_rbac_assignment BEFORE INSERT trigger checked user_profiles.organization_id = assignment.org_id — since organization_id was NULL, the check failed and the rbac_assignment INSERT was silently blocked. (c) The edge function only console.error'd the error without throwing, so the frontend received HTTP 200 even though no RBAC assignment was created.

Fix: Restructured accept action to (1) determine org_id first, (2) set it in the user_profiles upsert, then (3) insert the rbac_assignment.

Issue 3 — Permission toggles returned 404

Root cause: Same as Issue 1. check_is_org_super_admin was throwing on the missing public.users table reference, causing PostgREST to return 404 for DELETE on rbac_role_permissions.

Fix: Resolved by the same check_is_org_super_admin fix above.

Issue 4 — Permission changes required re-login

Root cause: The sync_rbac_to_jwt trigger only fired on rbac_assignments changes. Changes to rbac_role_permissions (i.e., the permission toggles in the Roles UI) never triggered a JWT update for affected users.

Fix: Added trg_sync_role_perms_to_jwt trigger on rbac_role_permissions that re-syncs the JWT for every user holding the affected role.

Issue 5 — RoleBadge.tsx crashed on new role codes

Root cause: roleConfig Record only contained the original legacy role codes. When a component received a new code like ADMIN or GM, roleConfig[role] returned undefined, and accessing .bgColor threw a TypeError.

Fix: Added all new role codes to roleConfig, ROLE_DEFINITIONS, and the UserRole type. Added fallbackConfig and makeFallbackDef helpers for future forward-compatibility.

Issue 6 — Settings/EHS/Assets invisible for ADMIN users

Root cause: isOrgAdmin() in permissions.ts only checked for ORG_ADMIN (legacy frontend code), missing the actual DB role codes ADMIN and GM.

Fix: Updated isOrgAdmin to check r.role === 'ADMIN' || r.role === 'GM' in addition to r.role === 'ORG_ADMIN'.

Issue 7 — Auth hydration race condition / 10-second login timeout

Root cause: AuthProvider ran checkAccountActive (a slow database RLS query) before calling hydrateFromSession, blocking setIsLoading(false) and causing the app to redirect to login on timeout.

Fix: Reversed the order — hydrateFromSession and setIsLoading(false) run first (synchronous, from JWT), then checkAccountActive runs as a non-blocking background check.

Issue 8 — Multi-tenant role code uniqueness blocked

Root cause: rbac_roles had a global UNIQUE(code) constraint that prevented a second organization from having a role named ADMIN.

Fix: Dropped rbac_roles_code_unique; the composite PRIMARY KEY (code, org_id) already enforces per-tenant uniqueness.
